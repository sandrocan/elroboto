\section{Embedded Platform and Software Architecture}
\label{sec:embedded_platform}

The STM32 integration had two principal requirements: the servo bus had to operate reliably at \(1\,\mathrm{Mbit/s}\), and application development had to remain separated from code generated by STM32CubeMX. The implementation therefore treats the CubeMX configuration as the hardware boundary and exposes the configured peripherals to handwritten application modules through narrow interfaces.

\subsection{Microcontroller and Communication Configuration}

The firmware targets the NUCLEO-U545RE-Q. Its system clock and the LPUART1 peripheral clock are both derived directly from the \(16\,\mathrm{MHz}\) high-speed internal oscillator without a phase-locked loop. LPUART1 uses a prescaler of one and is configured for \(1\,\mathrm{Mbit/s}\), 8-N-1 framing, bidirectional operation, and no hardware flow control. PA2 and PA3 provide the transmit and receive signals through alternate function AF8.

The explicit LPUART clock selection was essential: configuring the nominal baud rate alone is insufficient if the peripheral receives a different clock than assumed by the baud-rate generator. Using the \(16\,\mathrm{MHz}\) oscillator directly provides a defined clock-to-baud relationship and avoids an unnecessary clock-conversion stage in the motor communication path. Correct pin mapping and alternate-function selection were equally necessary before protocol-level debugging could produce valid servo responses.

Diagnostic output uses the independent ST-LINK virtual COM port at \(115200\,\mathrm{bit/s}\). Startup messages, faults, and controller telemetry therefore cannot be mistaken for binary servo packets, and diagnostic logging does not share the physical motor interface.

\subsection{Separation of Generated and Application Code}

CubeMX-generated startup, clock, GPIO, and peripheral initialization remain in the \texttt{Core} directory. The generated \texttt{main()} initializes the hardware, passes the LPUART1 handle to \texttt{App\_Init()}, and repeatedly calls \texttt{App\_Process()}. Handwritten code is kept in the \texttt{App} modules for UART access, servo communication, kinematics, numerical operations, control, and high-level sequencing. This boundary reduces manual changes to generated files and allows peripheral configuration to be regenerated without moving protocol or control logic into CubeMX user-code sections.

\texttt{App\_Init()} attaches the initialized UART handle, initializes the servo layer, and commands a defined starting pose. Encoder feedback is then checked against the start targets with a finite timeout. A failed command, communication error, aborted operation, or unreached start position transitions the application to a visible fault state rather than continuing from an unknown configuration.

\texttt{App\_Process()} implements the square demonstration. After a debounced button check, it reconstructs the initial tool center point through forward kinematics and generates four Cartesian waypoints in a constant-\(x\) plane. The selected motion routine must complete successfully before the waypoint index advances. The current resolved-rate routine is blocking, but its bus operations and complete motion have finite timeouts and an abort callback. Consequently, communication and convergence failures propagate to the application instead of being silently ignored.

\subsection{Integration for Controller Evaluation}

The application provides the common square geometry, servo speed and acceleration, timeout, joint limits, and inverse-kinematic parameters. Both the one-shot baseline and the resolved-rate controller use these same interfaces and differ only in how encoder feedback is used after the initial command. Synchronous position reads collect one joint-state set per cycle with less request overhead and acquisition skew than separate read commands; synchronous writes place all four targets in one bus packet. This common execution path made the controller comparison specific to the feedback strategy rather than to differences in peripheral configuration or waypoint generation.
