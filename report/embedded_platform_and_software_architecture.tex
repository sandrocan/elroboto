\section{Embedded Platform and Software Architecture}
\label{sec:embedded_platform}

The STM32 communicates with the daisy-chained Feetech servos over a
\(1\,\mathrm{Mbit/s}\) serial bus, including synchronous access to the four
active joints. In parallel, it receives E-Skin data during motion and provides
an independent diagnostic channel. The firmware separates CubeMX-generated
peripheral initialization from the handwritten application logic. After
startup, the initialized peripheral handles provide the interface through
which the application accesses the configured UART channels.

\subsection{Microcontroller and Communication Configuration}

The firmware targets the NUCLEO-U545RE-Q. LPUART1 uses PA2 and PA3 (AF8) at
\(1\,\mathrm{Mbit/s}\), 8-N-1, without flow control. Both the MCU and peripheral
clock derive from the \(16\,\mathrm{MHz}\) internal oscillator. Explicit clock
selection and the enabled RX FIFO provided reliable reception of complete servo
responses. During bring-up, raising the system clock from \(4\,\mathrm{MHz}\) to
\(16\,\mathrm{MHz}\) and receiving each known response as one block eliminated
the observed byte loss.

The E-Skin follows Bergner, Dean-Leon, and Cheng
\cite{bergner2020eskin}. An ESP32-C6 transmits the maximum proximity value from
GPIO16 to PC11/UART4\_RX at \(115200\,\mathrm{bit/s}\); UART4\_TX is unused.
The ST-LINK virtual COM port independently carries logs and controller
telemetry at the same baud rate.

\subsection{Generated Code and UART Processing}

The \texttt{.ioc} file defines clocks, GPIOs, interrupts, and peripherals. The
generated \texttt{main()} initializes both UARTs, passes their handles to
\texttt{App\_Init()}, and repeatedly calls \texttt{App\_Process()}. Servo,
UART, kinematics, control, and sequencing modules remain under \texttt{App};
the servo RX FIFO adjustment is confined to a CubeMX user-code section.

Servo requests use bounded blocking transfers with known response lengths.
Continuous E-Skin data is received byte-wise by interrupt. The callback accepts
a frame only when a newline follows five payload characters and then publishes
it through a sequence counter. Parsing and finite-value validation run later in
\texttt{UartCell\_Process()}, keeping the interrupt short. Counters for valid
and invalid frames, UART errors, and receive-restart failures support fault
analysis, while recovery remains deferred to foreground context.

\subsection{Application Integration and E-Skin Response}

\texttt{App\_Init()} starts E-Skin reception before servo motion and requires a
valid sample within \(2\,\mathrm{s}\). Missing data or an initial value of at
least \(0.050\) latches \texttt{FAULT}. After this check, the application moves
to and verifies its start pose before executing four Cartesian square
waypoints with the resolved-rate controller from
Section~\ref{sec:feedback_control}.

The blocking motion routine invokes the safety callback every control cycle.
At an E-Skin value of \(0.050\), it stops trajectory updates, commands the four
active joints to hold their measured positions, and enters
\texttt{PAUSED\_SKIN}. Motion resumes only after the value remains at or below
\(0.020\) for \(500\,\mathrm{ms}\). Stale data, proximity during startup, or
a failed hold command instead produces \texttt{FAULT}. This software response
is not a certified emergency stop: its latency depends on the current control
and UART operation, successful holding still requires communication, and the
servos remain powered. A physical means of disconnecting actuator power is
still required.

These STM32 motion thresholds are independent of the ESP32's visual-feedback
thresholds. The latter change the E-Skin LEDs at \(0.03\), \(0.016\), and
\(0.80\), but do not directly command robot motion.

\subsection{Control Integration}

The controller comparison used identical geometry, limits, motion parameters,
and timeouts. The one-shot baseline evaluated encoder feedback only after
standstill, whereas the selected controller repeatedly read all active joints,
reconstructed the TCP, calculated a bounded Cartesian correction, and
synchronously updated the targets. Detailed controller design and results are
given in Section~\ref{sec:feedback_control}.

The main integration challenge was combining continuous sensor traffic with
tightly spaced servo responses and a blocking motion interface. Moving parsing
out of the interrupt, enabling the servo RX FIFO, and reusing the motion abort
callback provided sufficient responsiveness without introducing an RTOS.
